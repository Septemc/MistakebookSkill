# Storydex 错题集能力原理说明
日期：2026-04-19

## 1. 文档定位

本文是对 Storydex 当前“错题集”能力的独立说明文档。

它的目标不是描述整个 Agent，而是单独抽出“纠错闭环 + 错题沉淀 + 后续回注”的这一项能力，方便后续：

1. 作为独立 Skill 提炼。
2. 移植到另一个项目中。
3. 作为记忆系统中的一个专用子模块复用。

本文全部内容基于当前 Storydex 代码中的真实实现整理，不是假设设计稿。

---

## 2. 一句话定义

Storydex 的错题集能力，本质上是一个：

`用户纠错确认 -> LLM 提炼规则 -> 项目级失败案例落盘 -> 后续检索可回注`

的闭环机制。

它不是单纯把“用户说你错了”记下来，而是把一次纠错过程转化成：

1. 可归档的案例
2. 可检索的规则
3. 可再次注入 Agent 上下文的经验

所以它更接近：

`项目级纠错知识沉淀器`

而不是普通“日志记录器”。

---

## 3. 这项能力解决的核心问题

在没有错题集能力之前，一个 Agent 常见的问题是：

1. 本轮纠错后，下轮又犯同类错误。
2. 用户指出的问题只停留在聊天记录里，不能结构化沉淀。
3. 纠错内容很多，但后续模型无法稳定读回。
4. “我已经理解了什么”和“以后必须遵守什么”没有被明确区分。

Storydex 当前的错题集能力，就是为了解决这四类问题。

它要做的不是“证明模型认错”，而是：

1. 把错误经验沉淀成项目资产。
2. 把纠错结果从对话层升级为记忆层。
3. 让历史纠错结果在未来任务中重新参与检索。

---

## 4. 当前能力边界

当前 Storydex 里的错题集能力，边界非常明确：

### 4.1 它负责什么

1. 允许用户开启“纠错模式”。
2. 在一次回答结束后强制要求用户标记“已完成 / 未完成”。
3. 在用户确认“已完成”后，调用 LLM 对这次纠错做结构化总结。
4. 将总结结果落盘到项目目录下的错题集。
5. 将这次错题沉淀同步写回会话历史。
6. 让错题文件后续能被 Agent 检索到。

### 4.2 它当前不负责什么

1. 不自动判断“这次输出有没有错”。
2. 不自动把错题升级成项目规则或技能。
3. 不自动修改已有规则文件。
4. 不负责多轮纠错案例的高级聚合。
5. 不负责跨项目共享错题。

所以当前这项能力已经是一个完整闭环，但仍是：

`人工触发 + 项目内归档 + 检索可回注`

的能力，而不是全自动学习系统。

---

## 5. 当前能力在系统中的位置

## 5.1 前端入口

主要位于：

1. `apps/frontend/src/components/AgentPanel.vue`
2. `apps/frontend/src/stores/agent.ts`

前端负责：

1. 显示“纠错”开关。
2. 在运行完成后弹出“请问我有没有纠错完成？”卡片。
3. 在用户未标记之前锁定输入框。
4. 用户点击“已完成 / 未完成”后更新本地状态，并决定是否发起归档接口请求。

## 5.2 后端入口

主要位于：

1. `apps/backend/api/routes_agent.py`
2. `apps/backend/services/story_project_service.py`

后端负责：

1. 接收 `/agent/correction-summary` 请求。
2. 调用 `summarize_correction(...)` 做结构化总结。
3. 把总结写入 `.storydex/.agent/failures/`
4. 更新 `INDEX.md`
5. 把 correction 信息写回 trace history。

## 5.3 后续回注入口

主要位于：

`apps/backend/agent/core/context_pipeline.py`

它会扫描：

`<project>/.storydex/.agent/failures`

并把其中部分文件作为 `agent_failure:` 上下文块参与后续检索。

这一步很关键，因为这意味着错题集不是“存档后就死了”，而是：

`归档后可再次参与推理`

---

## 6. 能力工作流总览

可以把这项能力理解成一个 6 步闭环：

### 第 1 步：用户显式开启纠错模式

前端 `agentStore.correctionModeEnabled` 决定当前轮是否进入纠错模式。

只有开启后，本轮运行才会被标记为：

1. `correctionMode = true`
2. `correctionReviewStatus = pending`

### 第 2 步：Agent 正常完成一轮运行

这一步本身与普通运行没有本质差异。

区别在于：

1. 当前 trace 会带上 correctionMode 状态。
2. 前端运行完成后，不会立刻完全释放输入逻辑，而是进入“等待用户确认纠错结果”的阶段。

### 第 3 步：系统弹出纠错确认卡片

当前界面文案为：

`请问我有没有纠错完成？`

用户可以选择：

1. `已完成`
2. `未完成`

这是这项能力最关键的人机交互节点。

### 第 4 步：如果用户点击“未完成”

系统行为是：

1. 当前 trace 的 `correctionReviewStatus` 变成 `incomplete`
2. 输入框重新可用
3. 输入提示变成：
   `请您为我提供更详细的纠错建议，本次纠错我哪里是正确处理，哪里是完全没有理解的？`

这个设计的意义是：

1. 不强迫系统在信息不足时归档错误结论。
2. 给用户一个追加高质量纠错说明的入口。
3. 将“用户继续补充纠错信息”纳入下一轮对话。

### 第 5 步：如果用户点击“已完成”

系统行为是：

1. 前端调用 `/agent/correction-summary`
2. 后端将本轮纠错材料发给 LLM
3. LLM 生成结构化总结
4. 总结写入项目错题集目录
5. 同步写回当前 trace history

### 第 6 步：后续任务中，错题可再次被检索

由于 `failures/` 已经被 Context Pipeline 扫描，所以未来相关任务中，这些错题文件有机会再次注入上下文，参与后续生成。

## 7. 当前前端交互原理

## 7.1 状态字段

前端以每条运行记录 `AgentExecutionRun` 为单位，维护两项关键状态：

1. `correctionMode`
2. `correctionReviewStatus`

其中 `correctionReviewStatus` 只有三种值：

1. `pending`
2. `completed`
3. `incomplete`

### 语义解释

1. `pending`
   - 本轮处于纠错模式，且等待用户确认本轮纠错是否完成。
2. `completed`
   - 用户确认本轮纠错完成，可以归档为错题总结。
3. `incomplete`
   - 用户认为系统还没纠错完成，需要继续追加更具体反馈。

## 7.2 输入框锁定机制

当前输入框会在某些情况下锁定。

和错题集直接相关的是：

1. 当本轮 `correctionReviewStatus = pending`
2. 且运行已经结束、没有报错

前端会把：

1. `correctionInputLocked = true`
2. 输入框 placeholder 改为：
   `请您先标记纠错结果`

这个机制的作用不是限制用户，而是强制闭环：

1. 先确认本轮纠错状态
2. 再继续下一步

它避免用户在“这一轮到底算没算纠错成功”尚未明确时，直接把会话推到下一轮，导致错题集沉淀失真。

## 7.3 “未完成”后的跟进机制

如果用户点击“未完成”，前端不会直接归档，而是：

1. 解锁输入
2. 标记 `correctionFollowupTraceId`
3. 用固定提示语引导用户补充更详细的纠错意见

这使错题集能力具备一个很实用的特征：

`先让用户把问题讲清楚，再归档`

这是比“立即归档失败记录”更稳妥的做法。

## 8. 当前后端归档原理

## 8.1 对外接口

当前归档接口为：

`POST /agent/correction-summary`

请求体核心字段包括：

1. `sessionId`
2. `traceId`
3. `originalPrompt`
4. `originalReply`
5. `correctionFeedback`
6. `finalReply`

### 这些字段的含义

1. `originalPrompt`
   - 这轮对话中用户的原始输入
2. `originalReply`
   - 这轮系统原始回复
3. `correctionFeedback`
   - 用户给出的纠错反馈内容
4. `finalReply`
   - 当前最终回复

注意：
从“理想语义”上看，`correctionFeedback` 应当是用户明确提供的纠错意见。
但在当前实现里，它有时会退化为当前 run 的 prompt 文本，因此这部分在能力迁移时应单独注意，不能机械照搬。

## 8.2 总结器的核心逻辑

真正的纠错总结发生在：

`StoryProjectService.summarize_correction(...)`

它会调用 LLM，并要求返回一个结构化 JSON，当前 schema 目标包括：

1. `title`
2. `summary`
3. `rules`
4. `confirmedUnderstanding`

### 这四项为什么重要

1. `title`
   - 便于索引与检索
2. `summary`
   - 提炼本次纠错的核心结论
3. `rules`
   - 提炼“以后必须遵守”的可执行规则
4. `confirmedUnderstanding`
   - 区分“我已经理解对了什么”，避免只记录负向错误

这四项合在一起，构成了错题集能力的最小有效信息单元。

## 8.3 归档文件生成逻辑

总结生成后，后端会写入：

`<project>/.storydex/.agent/failures/<timestamp>_<trace>.md`

文件内容当前大致包含：

1. 标题
2. 会话 ID
3. trace ID
4. 时间
5. 核心总结
6. 后续必须遵守
7. 已理解的点
8. 原始问题
9. 用户纠错反馈
10. 最终回答

也就是说，当前错题文件不是简单 JSON，而是一份：

`结构化 Markdown 案例文档`

它既适合人看，也适合后续检索。

## 8.4 索引文件更新逻辑

后端还会更新：

`<project>/.storydex/.agent/failures/INDEX.md`

索引只做一件事：

1. 给每条错题总结补一个目录入口

这让错题集具备了最基础的“目录页”能力，方便人类浏览，也方便项目管理。

---

## 9. 错题集落盘结构

当前最核心的目录是：

```text
<project>/.storydex/.agent/failures/
├─ INDEX.md
└─ 20260419TxxxxxxZ_<trace>.md
```

### 9.1 `INDEX.md` 的作用

1. 作为错题集目录
2. 记录每条总结的标题与链接
3. 提供人工浏览入口

### 9.2 单条错题文件的作用

1. 保存完整案例
2. 保存规则提炼结果
3. 保存“已理解内容”
4. 作为未来检索时的上下文材料

如果把这项能力迁移到别的项目里，建议保留这种“双文件结构”：

1. `INDEX`
2. `entry files`

因为它兼顾了：

1. 目录可浏览性
2. 单条案例的可引用性
3. 文本检索友好性

---

## 10. 错题集如何重新参与 Agent 推理

这是当前 Storydex 错题集能力里最关键的一点。

如果错题只是写盘，但后续永远不会被重新读回，那它只是文档，不是能力。

Storydex 当前已经把这一步接通了。

## 10.1 失败目录已加入 Context Pipeline

在 `AssistantContextPipeline._build_project_agent_blocks(...)` 中，`failures` 被作为一个正式的记忆来源扫描。

对应 source prefix 为：

`agent_failure:`

这意味着：

1. 错题文件会像 rule / skills / memory 一样参与候选上下文构建。
2. 它们不是外围附属物，而是 Agent 记忆的一部分。

## 10.2 参与方式仍然是“候选块”，不是强制注入

当前 Context Pipeline 的逻辑是：

1. 扫描文件
2. 截取预览
3. 根据 terms 和 route 做打分
4. 超预算则压缩或丢弃

所以错题集虽然已经进入检索体系，但当前仍然是：

`有机会被召回`

而不是：

`必定强制注入`

这意味着它已经具备“学习后可复用”的雏形，但还不是刚性规则系统。

## 11. 这项能力的最小状态机

如果要把它抽成一个独立 Skill，最推荐的抽象方式是直接保留状态机。

当前最小状态机可以表达为：

### 状态 1：`disabled`

未开启纠错模式。

### 状态 2：`armed`

已开启纠错模式，但本轮运行还未结束。

### 状态 3：`pending_review`

本轮运行结束，等待用户标记：

1. 已完成
2. 未完成

### 状态 4：`followup_needed`

用户认为未完成，等待补充更详细纠错建议。

### 状态 5：`summarizing`

用户标记已完成，系统调用 LLM 提炼错题总结。

### 状态 6：`archived`

错题总结已写入项目错题集。

## 12. 抽成独立 Skill 时，建议保留的最小接口

如果你要把它独立出来，最推荐保留以下 5 类接口。

## 12.1 模式控制接口

1. `enable_correction_mode()`
2. `disable_correction_mode()`

作用：
控制本轮会不会进入纠错闭环。

## 12.2 结果标记接口

1. `mark_correction_completed(trace_id)`
2. `mark_correction_incomplete(trace_id)`

作用：
显式切换本轮纠错状态。

## 12.3 归档接口

1. `summarize_correction_case(payload)`

输入建议至少包含：

1. 原始问题
2. 原始回答
3. 用户纠错反馈
4. 最终回答
5. session / trace 标识

输出建议至少包含：

1. title
2. summary
3. rules
4. confirmed_understanding
5. archive_path

## 12.4 存储接口

1. `append_failure_index(entry)`
2. `write_failure_entry(case)`

作用：
让错题集始终具备：

1. 目录页
2. 单条案例页

## 12.5 回注接口

1. `list_failure_memory_blocks(query, limit)`

作用：
让历史错题重新参与上下文构建。

如果没有这一层，错题集就不是“能力”，只是一堆 Markdown 文件。

## 13. 数据模型建议

如果换到另一个项目里，我建议保留如下抽象数据模型。

## 13.1 运行态模型

```json
{
  "traceId": "string",
  "correctionMode": true,
  "correctionReviewStatus": "pending|incomplete|completed"
}
```

## 13.2 归档请求模型

```json
{
  "sessionId": "string",
  "traceId": "string",
  "originalPrompt": "string",
  "originalReply": "string",
  "correctionFeedback": "string",
  "finalReply": "string"
}
```

## 13.3 归档结果模型

```json
{
  "failurePath": "string",
  "title": "string",
  "summary": "string",
  "rules": ["string"],
  "confirmedUnderstanding": ["string"]
}
```

## 13.4 单条错题案例推荐结构

```markdown
# 标题

- session: ...
- trace: ...
- time: ...

## 核心总结
...

## 后续必须遵守
- ...

## 已理解的点
- ...

## 原始问题
...

## 用户纠错反馈
...

## 最终回答
...
```

这种结构的优点是：

1. 对人类友好
2. 对文本检索友好
3. 对后续 LLM 总结也友好

## 14. 当前实现的优势

### 优势 1：闭环是完整的

它已经不是一个单点功能，而是完整闭环：

1. 触发
2. 确认
3. 补充
4. 总结
5. 写盘
6. 回注

### 优势 2：强调“用户确认”

当前设计不是系统自己判定是否纠错完成，而是让用户明确标记。

这在复杂创作类任务中非常重要，因为：

1. 正确与否常常是语义性的
2. 用户往往比系统更知道哪里没纠正到位

### 优势 3：归档结果不是只记录错误，而是同时记录“正确理解”

`confirmedUnderstanding` 这一项很有价值。

因为很多系统只会沉淀“错在哪里”，不会沉淀“现在理解对了什么”。
而后者对于长期稳定改进同样重要。

### 优势 4：与项目记忆体系天然兼容

它直接存到项目目录，并且已经被纳入 Agent 检索体系。

所以错题集不是孤岛，而是项目记忆网络的一部分。

## 15. 当前实现的限制与迁移时要注意的问题

## 15.1 当前归档触发仍然依赖人工点击

这是合理设计，但也意味着：

1. 没有用户确认就不会归档
2. 自动学习程度有限

如果在另一个项目中希望更自动化，需要额外引入自动判定机制。

## 15.2 当前 `correctionFeedback` 语义并不总是足够纯净

当前实现里，归档接口使用的 `correctionFeedback` 有时会回退到 run 的 prompt。

这意味着：

1. 它不一定总是“纯粹的用户纠错意见”
2. 在迁移时最好单独维护“用户纠错补充内容”字段

这是我认为当前实现里最值得注意的一个边界。

## 15.3 当前是“错题文件可回注”，不是“错题规则强执行”

现在错题集会参与检索，但不会自动变成：

1. 硬规则
2. 强校验器
3. 必选约束

迁移时如果你希望更强执行力，需要额外增加：

1. 规则提炼器
2. 规则同步器
3. 执行前校验器

## 15.4 当前多轮纠错聚合能力还不强

现在更偏向：

`单 trace 归档`

而不是：

`一整个纠错会话链归档`

如果另一个项目需要更强的训练价值，建议加入：

1. case-level 聚合
2. 多轮反馈合并
3. 单案例版本演进

---

## 16. 如果把它单独写成 Skill，我建议的能力定义

最推荐把这个 Skill 定义成：

### Skill 名称建议

`correction-memory`
或
`failure-codex`

### Skill 核心职责

1. 对一轮纠错结果进行结构化提炼。
2. 把提炼结果归档成项目级失败案例。
3. 从失败案例中提取可执行规则与正确理解点。
4. 在未来任务中按需将失败案例重新注入上下文。

### Skill 的最小组成

1. 纠错模式状态机
2. 纠错确认卡片或等价交互
3. 归档总结器
4. 失败案例存储层
5. 检索回注层

如果缺少第 5 项，这个 Skill 只能叫“纠错日志”；
只有把历史失败重新送回模型，它才真正成为“能力”。

---

## 17. 推荐的独立化抽象结论

如果你要把 Storydex 的错题集能力抽到另一个项目，我建议你把它当作下面这个能力来理解：

`一个带人工确认的、项目级的、可回注的纠错记忆子系统`

这句话里的每个词都很关键：

1. `带人工确认`
   - 不让系统自己武断归档错误结论
2. `项目级`
   - 错题集属于项目，而不是全局共享杂项
3. `可回注`
   - 归档不是终点，重新参与推理才是关键
4. `纠错记忆子系统`
   - 它不是 UI 附件，而是记忆体系的一部分

---

## 18. 最终总结

当前 Storydex 的错题集能力已经具备一个非常适合独立抽象的雏形：

1. 它有明确触发入口。
2. 它有清晰状态机。
3. 它有前后端闭环。
4. 它有稳定落盘结构。
5. 它能重新进入检索链路。

这意味着它完全可以被单独提炼成一个 Skill。

如果只保留一句最核心的抽象，我建议写成：

`错题集不是“记录错误”，而是把用户纠错转化为可检索、可复用、可再次约束模型行为的项目级经验。`
