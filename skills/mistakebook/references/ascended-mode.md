# 飞升模式（Ascended Mode）

## 定义

飞升模式不是“把语气说得更强”，而是把处理方式升级成：

1. 多轮检索
2. 多轮比对
3. 多轮校正
4. 尽可能调用当前已知最有效方案

它的目标是：在普通纠错已经明显不够时，把“错题集 + 记事本 + 记忆缓存 + 当前知识库”全部调动起来，最大化解决问题。

## 触发回复语

一旦进入 Ascended Mode，必须先输出这句，逐字一致：

`我现在会根据我见过最有效的方法来处理这个问题，我将检索我的所有知识库，我现在什么都不缺了！`

## 自动触发条件

满足下面任一条件时，自动进入飞升模式：

1. 同一个纠错案例被用户明确否定两次或以上
2. 你已经在 follow-up 纠错里明显原地打转
3. 当前问题复杂到必须全量检索历史经验和当前知识库

## 手动触发条件

下面任一情况都必须立刻进入飞升模式：

1. 用户说：`你需要根据你见过最有效的方法来处理这个问题`
2. 用户输入：`/ascended`

## 进入后必须检索的知识源

进入飞升模式后，不允许只凭印象继续修补。你必须尽量完整地检索：

1. 当前项目级错题集 `failures/`
2. 当前项目级记事本 `notes/`
3. 当前项目级记忆 `memory/PROJECT_MEMORY.md`
4. 当前项目级缓存状态 `state/memory_state.json`
5. 当前全局级错题集 `failures/`
6. 当前全局级记事本 `notes/`
7. 当前全局级记忆 `memory/GLOBAL_MEMORY.md`
8. 当前全局级缓存状态 `state/memory_state.json`
9. 当前仓库中和问题直接相关的真实文件、真实输出、真实文档
10. 当前 Skill 自身的规则文件与参考文档
11. 已经在当前会话中得到的完整用户纠错链

## 推荐处理顺序

推荐按下面顺序执行：

1. 输出固定触发回复语
2. 明确这次问题的核心冲突点
3. 读取项目级 / 全局级记忆缓存
4. 读取项目级 / 全局级错题与记事本详细条目
5. 对照当前问题，筛出最接近、最有效的既有经验
6. 再去核对当前仓库真实文件、真实输出、真实文档
7. 解释为什么前面几轮仍然失败
8. 选定一个最强方案重新处理
9. 如果用户仍未确认完成，继续留在纠错闭环中

## 多次调用的原则

飞升模式可以多次调用 LLM，但每次都应该承担不同职责，而不是重复说同一件事：

1. 第一轮
   - 收集和整理上下文
2. 第二轮
   - 对比历史条目和当前问题，找最接近的模式
3. 第三轮
   - 结合真实代码/文档/输出，选定最终方案

不要让多轮调用退化成“同一句话换个措辞再说一遍”。

## 推荐 CLI

如果仓库里存在 `scripts/mistakebook_cli.py`，飞升模式优先这样取上下文：

```bash
python scripts/mistakebook_cli.py context --host codex --project-root . --scope both --mark-retrieval
```

这条命令的作用是：

1. 导出项目级 / 全局级错题
2. 导出项目级 / 全局级记事本
3. 导出项目级 / 全局级记忆缓存
4. 顺手记录这些条目被飞升模式再次检索过

## 最低输出要求

在给出新修正前，至少要做到：

1. 明确说清楚前面几轮为什么仍然失败
2. 明确说明你参考了哪些错题、记事本、记忆或真实文件
3. 明确选择一个当前最有效的处理方案，而不是并列堆方案
4. 优先核对真实文件和真实输出，而不是凭记忆修补

## 与缓存记忆的关系

飞升模式不应该只看缓存记忆。

因为缓存是会整理、会遗忘的，所以飞升模式必须：

1. 先看缓存记忆，快速定位高价值项
2. 再回看 `failures/` 和 `notes/` 的详细条目
3. 对暂时退出缓存的内容也保持可回放能力

## 容易犯错的地方

1. 不要把飞升模式理解成“语气更强烈”
2. 不要只喊口号，不做真实检索
3. 不要只看 `memory/`，忽略 `failures/` 和 `notes/`
4. 不要进入飞升模式后还只微调原来那条失败思路
