# Ascended Mode

## 定义

Ascended Mode 是“错题集 Skill”的高级能力。

当普通纠错已经进行了两次以上、用户仍然认为你没有改对，或者用户明确要求你用“你见过最有效的方法”来处理问题时，不能再停留在浅层修修补补，而是要进入一次真正的全量检索和深度分析。

## 触发回复语

一旦进入 Ascended Mode，必须先输出这句，逐字一致：

`我现在会根据我见过最有效的方法来处理这个问题，我将检索我的所有知识库，我现在什么都不缺了！`

这句是神级模式的固定触发回复语。对同一个 case，只在真正进入 Ascended Mode 的时候展示一次。

## 自动触发条件

同一个纠错案例里，满足下面条件之一时，自动进入 Ascended Mode：

1. 用户已经让你改了两次或以上，仍然明确说不对
2. 同一个 case 的 `rejection_count >= 2`
3. 你已经在 follow-up 纠错里出现明显原地打转

这里的“仍然说不对”通常包括：

1. `还不对`
2. `你还是没理解`
3. `我都纠正你两次了`
4. `你又错了`
5. `还是有问题`

## 手动触发条件

下面任一情况都必须立刻进入 Ascended Mode：

1. 用户说：`你需要根据你见过最有效的方法来处理这个问题`
2. 用户输入：`/ascended`

## 进入后必须做什么

进入 Ascended Mode 后，不允许只凭印象继续修补。你必须系统性检索并整合下面这些知识来源：

1. 当前项目级错题集 `failures/`
2. 当前项目级记忆 `memory/PROJECT_MEMORY.md`
3. 当前全局级错题集 `failures/`
4. 当前全局级记忆 `memory/GLOBAL_MEMORY.md`
5. 当前仓库中和问题直接相关的真实文件、真实输出、真实文档
6. 当前 Skill 自身的规则文件与参考文档
7. 已经在当前会话中得到的用户纠错链

## 进入后的处理顺序

推荐按这个顺序执行：

1. 输出固定触发回复语
2. 明确本次问题的核心冲突点
3. 全量检索项目级与全局级错题/记忆
4. 对照当前问题，找出最接近、最有效、曾经成功的处理模式
5. 重新分析这次为什么会连续纠错仍失败
6. 基于最强方案重新给出修正
7. 如果用户仍未确认完成，继续留在纠错闭环并追加固定追问句

## 进入后最低输出要求

在给出新修正前，至少要让你的思路满足下面要求：

1. 明确说清楚这次连续纠错失败的根因
2. 明确说明你参考了哪些项目级/全局级知识来源
3. 明确选择一个“当前最有效”的处理方案，而不是并列堆方案
4. 如果涉及代码、配置、文档或规则，优先核对真实文件和真实输出

## 与普通纠错模式的关系

Ascended Mode 不是替代纠错模式，而是纠错模式内的高级升级。

也就是说：

1. 进入 Ascended Mode 后，case 仍然是同一个 case
2. 用户没有确认完成前，依然要维持固定追问句
3. 只有用户明确确认完成后，才能归档

## 归档时建议补充的字段

为了让后续回放更有价值，建议在归档 payload 里增加：

1. `correctionAttemptCount`
2. `ascendedTriggered`
3. `ascendedTriggerReason`
4. `knowledgeSourcesReviewed`

## 容易犯错的地方

1. 不要把 Ascended Mode 理解成“语气更强烈”
2. 不要只喊口号，不做真实检索
3. 不要进入神级模式后又只微调同一思路
4. 不要忽略项目级记忆与全局级记忆的差异
