# Activation Patterns

## 应该触发的场景

下面这些表达通常意味着用户正在纠正你：

1. `你这里错了`
2. `这不对`
3. `你又犯同样的错`
4. `重新改`
5. `按我说的改`
6. `我来纠正你`
7. `我教你一遍`
8. `还没改对`
9. `你没有吃透`
10. `你理解反了`
11. `这个回答有问题`
12. `这个地方你改坏了`
13. `你又把之前的要求漏掉了`
14. `启动错题集`
15. `纠正`
16. `还是有问题`
17. `请重新改`
18. `你改的有问题`
19. `/mistakebook`
20. `你需要根据你见过最有效的方法来处理这个问题`
21. `/ascended`

## 不应该触发的场景

下面这些情况不要自动进入错题集闭环：

1. 用户在讨论业务问题本身，而不是纠正你
2. 用户说“这个程序报错了”，但没有指出你的回答错误
3. 用户在做普通 review，没有表现出纠错意图
4. 用户只是要求你继续下一步，而不是回头纠正你
5. 用户在引用别人说的话，不是在纠正你

## 进入后的第一步

一旦确定触发：

1. 先输出激活句
2. 再执行修正
3. 结尾追加固定追问句

不要先解释流程，再说激活句；激活句本身就是流程开始信号。

## 继续纠错的判断

如果用户后续说：

1. `还不对`
2. `你再看一遍`
3. `我不是这个意思`
4. `你还是没理解`
5. `我再教你一次`

把它们视为同一个案例的 follow-up，而不是新建案例。

同时维护 `rejection_count`。

如果同一个案例里，用户已经连续两次以上明确表示“还不对 / 还是错 / 还是没理解”，就不要再停留在普通纠错模式，而要自动升级到 Ascended Mode。

## 神级模式触发

下面任一情况都应该进入 Ascended Mode：

1. 同一个纠错 case 的 `rejection_count >= 2`
2. 用户明确说：`你需要根据你见过最有效的方法来处理这个问题`
3. 用户输入：`/ascended`

进入后先输出固定神级回复语，再进行项目级 / 全局级错题与记忆的全面检索。

## 完成纠错的判断

以下表达可视为显式完成信号：

1. `可以了`
2. `这次对了`
3. `已经纠正好了`
4. `已经吃透了`
5. `归档吧`
6. `写入错题集`
7. `完成纠错`

一旦收到，停止追加固定追问句，转入归档。

## 误触发防线

在下面两类情况里，优先保守处理：

1. 用户语义不清
   - 先做一次小修正，再在结尾继续问固定追问句
2. 用户只是在表达不满意，但没有明确在纠正
   - 可以继续修正，但不要立即归档，等显式完成信号
